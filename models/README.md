# models/ — Isolation Forest + XGBoost training, validation, and pipeline integration notes

This folder holds both models: the unsupervised anomaly-detection model
(Isolation Forest, trained on benign-only multi-cloud IAM logs) and the
supervised multi-class attack-type classifier (XGBoost, trained on labeled
attack data), plus the scripts that produce and validate each. SHAP wiring is
still not done — see "Next steps" below.

## Dataset splits and what each is allowed to touch (read before tuning)

| Split | Dir | Generated by | Role — what it may inform |
|---|---|---|---|
| Train (benign) | `Datasets/Train_iso` | `generate_benign --split train` | Isolation Forest training only |
| Holdout (benign) | `Datasets/holdout` | `generate_benign --split holdout` | benign half of the **final eval** set |
| Attacks (fast) | `Datasets/attacks_fast` | `generate_attacks --seed 7 --pace fast` | XGBoost training; leaks into any XGBoost decision |
| Attacks (slow) | `Datasets/attacks_slow` | `generate_attacks --seed 13 --pace slow` | attack half of the **final eval** set — report once, never tune on it |
| **Attacks (cal)** | `Datasets/attacks_cal` | `generate_attacks --seed 21 --env-seed 42 --pace slow` | **[Phase 0]** attack half of the **tuning/calibration** set |
| **Benign (cal)** | `Datasets/holdout_cal` | `generate_benign --seed 42 --split cal` | **[Phase 0]** benign half of the **tuning/calibration** set |

**Why the cal splits exist (Phase 0):** `attacks_fast` leaks into anything involving
XGBoost, and `attacks_slow` is the frozen final test set — there was no clean third split
to *make* tuning/threshold decisions on. The calibration set = `holdout_cal` + `attacks_cal`
(2291 + 369 = 2660 events, 9.9% attack base rate) fills that gap:

- `attacks_cal` — `generate_attacks --seed 21 --env-seed 42 --pace slow --noise-ratio 0.4
  --repeats 8 --days 21`. Same `--env-seed 42` population and `--pace slow` as `attacks_slow`
  (the open problems — graph regression, threshold fragility — only manifest on slow pacing),
  different activity seed (21 vs 13). Structurally matches `attacks_slow`: 369 events / 264
  attacks, identical `threat_category` distribution, 113 vs 114 distinct actors.
- `holdout_cal` — `generate_benign --days 21 --seed 42 --split cal`. The new `cal` split
  reuses the **same holdout population** (so its benign risk-score distribution matches the
  eval-benign set) but re-seeds `env.rng` (population fixed by `--seed`, activity stream
  offset by `CAL_STREAM_OFFSET`) so its events are **disjoint** from `holdout`. Verified:
  90/90 principals shared, 0 event-ID overlap, 2291 vs 2363 events.

From Phase 0 on: **all tuning/calibration decisions are made on the cal set**; `attacks_slow`
+ `holdout` numbers are reported once per phase as a frozen check, never used to pick a
parameter.

## Where this fits in the pipeline

```
raw cloud logs (AWS/Azure/GCP)
        |
app/parser_normalizer/src/pipeline.py  (ParserPipeline)          <- per-event normalize
        |
app/parser_normalizer/src/feature_extractor.py (MLFeatureExtractor)  <- BATCH feature
        |                                                            engineering, used
        |                                                            for offline training
   models/train_isolation_forest.py  -->  models/iso_forest.pkl
   models/validate_and_pick_threshold.py (adds decision_threshold to the bundle)
        |
        v  (live scoring path, not yet wired to iso_forest.pkl -- see Known gaps)
app/services/ml_inference.py (ParallelMLEngine)   <- per-EVENT scoring:
        |                                             anomaly_score, predicted_phase,
        |                                             shap_attributions
        v
app/services/graph_engine.py (MultiCloudGraphEngine)  <- cross-cloud IDENTITY STITCHING:
        |                                                  resolves which entity an event
        |                                                  belongs to (federation join ->
        |                                                  creation provenance -> fuzzy
        |                                                  UA/proxy/IP fusion), maintains a
        |                                                  60s sliding window PLUS a
        |                                                  permanent lifetime cloud
        |                                                  footprint per entity (so a
        |                                                  slow-paced attacker crossing
        |                                                  clouds minutes apart is still
        |                                                  detected as cross-cloud).
        v
app/services/risk_engine.py (RiskEngine)   <- per-ENTITY risk score: time-decayed sum of
                                               each active event's anomaly_score, multiplied
                                               by a per-new-cloud-crossed factor. Squashed
                                               to 0..1, alerts when it crosses a threshold.
```

**Important**: `graph_engine.py` and `risk_engine.py` are downstream consumers
of the model's output, not part of feature engineering. They operate on
already-scored events (each event dict needs `anomaly_score` attached by
`ml_inference.py` first) plus already-normalized fields (`principal`,
`ua_family`, `is_known_proxy_or_tor`, `source_cloud`, `timestamp`, etc.) — they
don't consume `MLFeatureExtractor`'s engineered 24-column feature matrix at
all. This means **the plain per-event Isolation Forest PR-AUC (see Results
below) is not the full system's detection performance** — it's the
first-layer signal that `risk_engine.py`'s cross-cloud multiplier is designed
to amplify for kill chains that span clouds. A proper system-level evaluation
needs to run scored events through `graph_engine.process_event()` and
`risk_engine.calculate_intensity()` and evaluate `is_critical`/`scaled_score`
per entity, not `anomaly_score` per raw event. Not done yet — see Next steps.

## How to run

**1. Train** (benign-only — never point this at data containing attacks):
```bash
python models/train_isolation_forest.py --train-dir Datasets/Train_iso --out models
```

**2. Validate + pick a threshold** (uses labeled holdout + one attack split; the *other* attack split must stay untouched as a final test set):
```bash
python models/validate_and_pick_threshold.py \
    --model models/iso_forest.pkl \
    --holdout-dir Datasets/holdout \
    --attack-dir Datasets/attacks_fast
```
`--metric f1` (default) picks the best-F1 point; `--metric precision_floor --precision-floor 0.9` instead picks the highest-recall point that still hits a precision floor you choose.

Regenerating the underlying datasets (if you need fresh ones):
```bash
cd dataset_script
python generate_benign.py  --days 21 --seed 42 --split train   --out ../Datasets/Train_iso
python generate_benign.py  --days 21 --seed 42 --split holdout --out ../Datasets/holdout
python generate_attacks.py --seed 7  --env-seed 42 --pace fast --noise-ratio 0.4 --repeats 8 --days 21 --out ../Datasets/attacks_fast
python generate_attacks.py --seed 13 --env-seed 42 --pace slow --noise-ratio 0.4 --repeats 8 --days 21 --out ../Datasets/attacks_slow
```
`--env-seed` must match across benign and attack calls (population identity), or the anti-leakage design breaks. `dataset_script/build_dataset.py` runs all four in one shot if you don't need the custom `--out` paths.

**3. See exactly which logs got flagged, and get a fuller metrics report:**
```bash
python models/inspect_flagged_events.py
```
Writes `models/scored_events.csv` (every event, sorted by anomaly score
descending) and prints confusion matrix / accuracy / ROC-AUC / FPR / a
per-threat-category and per-cloud breakdown. Run step 2 (`validate_and_pick_threshold.py`)
at least once first — this script reads `decision_threshold` from the bundle.
Point `--attack-dir Datasets/attacks_slow --out-csv models/scored_events_test.csv`
at it once (and only once) you're done tuning, to get the real held-out test number.

**4. Train + evaluate the XGBoost attack-type classifier:**
```bash
python models/train_xgboost.py --train-dir Datasets/attacks_fast --test-dir Datasets/attacks_slow --out models
```
Trains on `attacks_fast` (labeled attacks + benign noise mixed — **not**
benign-only data, which has no class diversity for a classifier to learn
from; that's what the Isolation Forest is for), evaluates on `attacks_slow`.
Defaults to `--device cuda` (RTX 3050); pass `--device cpu` if no GPU is
available.

## XGBoost: supervised multi-class attack-type classifier

Target: `threat_category` (`AccountCompromised`, `BruteForce`,
`CredentialCreation`, `DefenseEvasion`, `Normal`, `PrivilegeEscalation`,
`Reconnaissance`, `SuspiciousLogin`, `SuspiciousServiceAccountCreation`,
`UnauthorizedAccessAttempt`) — 10 classes, run in parallel with the Isolation
Forest per `ml_inference.py`'s existing design (every event gets both an
`anomaly_score` and a `predicted_phase`).

**Feature set differs from the Isolation Forest's 24 columns** — 28 columns,
via `feature_extractor.py`'s new `include_labeled_only_features=True` flag.
`geo_country`, `is_known_proxy_or_tor`, `device_compliant_status`, and
`is_internal_ip` were dropped for Isolation Forest training because they're
zero-variance on *benign-only* data (see "Known feature-engineering gaps"
below) — but they have real, confirmed variance on labeled attack data (e.g.
`is_known_proxy_or_tor` is 41 True / 328 False on `attacks_fast`, `geo_country`
has 16 distinct real country codes, `device_compliant_status` splits
32 `NonCompliant` / 34 `Compliant`). `user_id` stays dropped either way
(identity leakage — arguably worse for a classifier trained on a small
number of distinct campaign instances), and so do `browser_type` /
`is_new_ip_for_user` (confirmed still zero-variance / still a generator
artifact even on attack data, not fixed by adding labels).

**A second instance of the row-alignment bug was found and fixed while
building this** — the same class of bug documented below, reintroduced by
not reusing the existing fix consistently. The first version of
`train_xgboost.py` called `extractor.extract_features()` directly on the
training data instead of going through `scoring_utils.featurize_and_align()`
(which was only used for the test set). Since `extract_features()` sorts
rows by timestamp internally, `X_train`'s rows ended up in a different order
than `y_train` (built in original file-load order) — training on scrambled
label/feature pairs. Symptom was obvious once looked at: `Normal`, the
single largest class (28% of the data), scored exactly 0.00 precision AND
recall. Fixed by generalizing `featurize_and_align()` to support training
mode (`is_training=True, feature_columns=None`) so there's only one correct
code path, used for both train and test — see its docstring, which now says
explicitly not to bypass it.

**Results on `Datasets/attacks_slow` (held out, never touched until this
evaluation)**: accuracy 0.9106, macro F1 0.9019. Every class gets real
precision and recall — no class is missed the way `Normal`/`SuspiciousLogin`/
etc. were under the alignment bug. Full per-class report and confusion
matrix are in the script's output; rerun to see them again.

**Caveat worth being precise about in the paper**: checked feature
importances before trusting this number. `action` (the literal API call
name — closest thing to a direct answer key in a scripted synthetic
dataset) is *not* dominant, ranking 5th at 0.05 importance; top features are
`login_result_success` (0.28), `unique_resources_accessed` (0.27),
`mfa_authenticated` (0.10) — genuine behavioral/velocity signals. But
`login_result_success` and other per-step attributes are *also*
deterministic outputs of `attack_scenarios.py`'s fixed step templates, not
just `action`. `attacks_fast` and `attacks_slow` are both generated from the
exact same 4 scenario definitions (`--seed 7` vs `--seed 13` only changes
victims/timing/pacing, not which techniques exist or their scripted
attributes). So 0.91 accuracy mostly demonstrates "the model recognizes
which of these 4 known scripted scenarios a step belongs to," not "the
model generalizes to a genuinely novel attack technique." That's a real but
narrower claim — state it precisely rather than citing the headline number
alone. A leave-one-scenario-out test (train on 3 scenarios, test attack-type
classification on the 4th, never-seen one) would directly measure the
generalization claim if that's needed for the paper.

### Generalization (leave-one-scenario-out) — [Phase 3]

`models/evaluate_loso.py` runs it. **Framing:** true LOSO can't measure
*multi-class* accuracy on the held-out scenario (its `threat_category` labels may
be absent from the training folds — `BruteForce` and
`SuspiciousServiceAccountCreation` live only in the APT scenario), so the honest
metric is **binary generalization**: does `1 - P(Normal)` rank the unseen
scenario's attack events above benign? (That's also exactly the signal the risk
engine consumes.) Each fold trains XGBoost — same hyperparameters as
`train_xgboost.py`, via `featurize_and_align` — on the other 3 scenarios' attack
events + benign-noise, and tests on the held-out scenario's attacks + a disjoint
`holdout` benign sample. The scenario identifier is joined by `event_id` from
`attack_labels.csv` (already present — no dataset regeneration).

| Held-out scenario | binary ROC-AUC | PR-AUC | unseen flagged not-Normal |
|---|---|---|---|
| cross_cloud_apt_credential_theft | 0.999 | 0.976 | 99% |
| logging_tamper_and_destruction | 0.973 | 0.560 | 100% |
| service_account_key_abuse | 0.997 | 0.920 | 100% |
| **insider_privilege_abuse** | **0.515** | 0.038 | **2%** |
| **MEAN** | **0.871** | 0.623 | |

**State this next to the 0.91 in-distribution accuracy.** The model generalizes
to 3 of 4 unseen scenarios (ROC 0.97–0.999) — their events share behavioral
signals (failed logins, cross-cloud/Tor infra, velocity) with the training
scenarios, and the descriptive multi-class buckets show them landing on *other
attack* classes (the qualitatively better failure). The exception is decisive:
**`insider_privilege_abuse` collapses to ROC 0.515 (~random), 47/48 events
classified `Normal`.** An insider abusing already-valid privileges (successful
logins, MFA satisfied, compliant device, benign geo) *is behaviorally
indistinguishable from benign* to this feature set, and with that scenario held
out the model has no signal for it. So the defensible claim is: strong
generalization to unseen scenarios that carry overt attack signals, and a genuine
blind spot for low-and-slow insider abuse — NOT a blanket "generalizes to novel
techniques." `models/loso_results.csv` has the full per-fold breakdown.

## A correctness bug was found and fixed — read this before trusting any earlier number

`MLFeatureExtractor.extract_features()` sorts its output by global timestamp
and resets the row index (`feature_extractor.py`:
`df.sort_values('timestamp').reset_index(drop=True)`). The first version of
`validate_and_pick_threshold.py` built `y_true` by iterating the raw JSON
files in cloud-grouped order (all AWS events, then all Azure, then all GCP)
and zipped it against `X` by raw list position — but `X`'s row order after
`extract_features()` is a chronological interleaving of all three clouds, not
cloud-grouped. **The labels and the feature rows were never actually
aligned.** Confirmed empirically (a 15-row cloud-grouped test input came back
from `extract_features()` in a completely different order). The originally
reported PR-AUC (0.0998) was computed against misaligned labels and should be
disregarded in favor of the numbers below.

Fixed in `scoring_utils.py`: every event gets a unique `__eval_row_id__`
before featurization, which survives `extract_features()` untouched (it's not
in any drop/encode list) and is used afterward to realign `X` back to the
original input order via `argsort`. Both scripts now go through
`scoring_utils.featurize_and_align()` — don't zip labels against a fresh
`extract_features()` call by position anywhere else in this codebase, or this
bug will silently reappear.

## Current results (corrected — last run, rerun and update after any data/feature change)

Trained on `Datasets/Train_iso` (15,250 benign events, 24 engineered features).
Validated on `Datasets/holdout` + `Datasets/attacks_fast` (2,732 events, 264
genuine attack events, 9.66% base rate — `attacks_slow` intentionally not
touched, reserved as the final test set):

- **PR-AUC: 0.0926** (base rate 9.66%) — barely above a no-skill classifier.
- **ROC-AUC: 0.5229** — essentially coin-flip (0.5 = random). This is the
  more damning number: unlike precision/recall, ROC-AUC doesn't depend on
  threshold choice, so this says the model has almost no intrinsic ability to
  rank attack events above benign ones at the per-event level.
- Best-F1 operating point: threshold `-0.0776` on `-decision_function()`,
  precision 0.113, recall 0.867, F1 0.200.
- **Confusion matrix at that threshold**: TP=229, FP=1794, TN=674, FN=35.
  Accuracy 0.331, **False Positive Rate 0.727**. The "high recall" numbers
  below are a degenerate artifact of this: the best-F1 threshold is so low it
  flags ~73–78% of *everything* (benign included), so it trivially catches
  most real attacks too, at enormous false-positive cost. Looking at
  `scored_events.csv`'s top-ranked rows confirms this directly — the
  "most anomalous" events are mundane benign admin actions (`ConsoleLogin`,
  `AssumeRole`, `ListUsers`) by random legitimate users, nothing distinctive.
- **Detection rate by threat_category** (of real attack events, % flagged):
  SuspiciousServiceAccountCreation 100%, AccountCompromised 94%,
  CredentialCreation 91%, PrivilegeEscalation 89%, BruteForce 88%,
  Reconnaissance 88%, UnauthorizedAccessAttempt 88%, DefenseEvasion 83%,
  SuspiciousLogin 63% — all high, but meaningless on their own given the FPR
  above; read them only alongside it.
- **False positive rate by cloud**: AWS 0.784, GCP 0.730, AZURE 0.656 —
  roughly consistent across clouds, so this isn't a single-cloud data-quality
  artifact; it's a genuine property of the per-event feature set.

**Interpretation**: this is a clean, honest "before" baseline — the per-event
Isolation Forest genuinely cannot separate these classes well on its own
(ROC-AUC ~0.52 confirms this isn't just a threshold-tuning problem).

## Ablation table: does graph + risk scoring actually help?

`models/evaluate_full_pipeline.py` scores the same validation set
(`Datasets/holdout` + `Datasets/attacks_fast`) three ways, same underlying
anomaly scores throughout. Both B and C calibrate `alert_threshold` from that
row's own benign `risk_intensity` values (99th percentile, via
`RiskEngine.recalibrate_threshold()`) rather than trusting the uncalibrated
default 0.75 — same lesson as the IF `contamination` fix earlier.

| Row | ROC-AUC | PR-AUC | Precision @ threshold | Recall @ threshold | TP / FP |
|---|---|---|---|---|---|
| A. IF only (per-event) | 0.5229 | 0.0926 | 0.113 @ best-F1 | 0.867 @ best-F1 | — |
| B. IF + risk, no identity stitching | 0.5203 | 0.0920 | — | 0.000 | 0 / 25 |
| C. IF + risk + graph (full pipeline) | **0.5684** | **0.2297** | **0.615** | 0.152 | 40 / 25 |

Row C's numbers above are **after** three further tuning fixes on top of the
two bug fixes described below (each verified by rerunning
`evaluate_full_pipeline.py`, not assumed):
- `enrichment.py`'s `UA_PATTERNS` didn't recognize `curl`/`python-requests`/
  `python-urllib3`/`Go-http-client` — the exact tooling most attack traffic
  in this dataset uses — so they all fell through to `ua_family="Other"`,
  which the bug fix below correctly excludes from earning similarity credit.
  Net effect: the highest-weighted signal (0.45) contributed nothing for
  most attack traffic. Added real patterns for all four.
- `graph_engine.py` compared raw cloud-native `principal_type` strings
  (AWS `"IAMUser"`/`"AssumedRole"` vs Azure/GCP `"User"`/`"ServiceAccount"`),
  so a genuine human attacker's own AWS leg never earned type-match credit
  against their own Azure/GCP legs of the same campaign — pure vocabulary
  mismatch. Normalized into the same human/automation concept
  `risk_engine.py`'s `_classify_principal` already uses.
- `risk_engine.py`'s `half_life_seconds` defaulted to 30s, which only fits
  `generate_attacks.py`'s "fast" pacing (5-90s gaps). At a "slow"-pace 600s
  gap, a 30s half-life decays the previous step's contribution by `0.5**20`
  (~1e-6) — effectively zero, making the decayed-sum term blind to
  anything but the single most recent event for any campaign paced slower
  than ~2 minutes. Raised to 1800s (30 min): a 90s fast-pace gap still
  barely decays (~97% retained), while a 5400s (1.5h) slow-pace gap retains
  ~12.5% instead of ~0%. Benign traffic is unaffected either way (events
  average ~18h apart, decaying to ~0 under both settings).

Combined effect of these three tuning fixes on top of the bug fixes: ROC-AUC
0.5077 → 0.5684, PR-AUC 0.2036 → 0.2297, precision 0.545 → 0.615, recall
0.114 → 0.152, TP 30 → 40. Consistent improvement across every metric, not a
tradeoff between them — a good signal the fixes address real defects rather
than just shifting the operating point.

Row B is (correctly) an unchanged monotonic rescale of row A for
single-event entities. Row C is **after** the `graph_engine.py` fixes below —
before them it scored **ROC-AUC 0.3748 (worse than random)**.

### Two real bugs found and fixed in `graph_engine.py`

**1. Tier-2 fuzzy-fusion rewarded matching on ignorance, not identity.**
Diagnosed by running `MultiCloudGraphEngine` over `attacks_fast` (369 events,
148 distinct real identities) and logging entity resolution: only **3**
entities were created total — the engine wasn't failing to stitch genuine
cross-cloud attacker sessions (first hypothesis, wrong), it was massively
*over*-merging unrelated people. `_calculate_fuzzy_similarity` had three
instances of the same defect — a comparison meant to require two sides
*genuinely matching on a specific value* instead fired whenever both sides
simply *lacked information*:
- UA family: guarded against the literal string `"Unknown"`, but
  `enrichment.py`'s `parse_user_agent()` fallback returns `"Other"` for
  anything unrecognized (`curl`, `python-requests`, etc. — most attack
  traffic here) — `"Other" == "Other"` slipped past the guard and earned the
  full 0.45 weight.
- UA version: no guard at all — two `"Unknown"` versions matched and earned
  the extra 0.30 sub-bonus on top.
- Proxy flag: `proxy_a == proxy_b` fired for `(False, False)` too, awarding
  +0.125 to nearly every pair, since most traffic (benign and, thanks to the
  hosting/foreign ASN-keyword gap, much attack traffic too) isn't flagged.

Two unrelated people could clear `tau=0.65` on nothing but shared ignorance:
0.45 + 0.125 + 0.20 (type match) = 0.775. Fixed by only awarding credit when
both sides match on a genuinely informative value (excluding `Unknown`/
`Other`/empty from every comparison), and removing the False/False proxy
credit entirely (only a confirmed True/True Tor/hosting match is
informative). Also added a `tier2_lookback_seconds` bound (default 3h,
derived from `generate_attacks.py`'s slow-pace max gap of 1.5h with 2x
margin) so entities idle longer than that are no longer eligible Tier-2
candidates — previously the scan compared against *every* entity ever
created with no time relationship at all.

**2. Entity IDs collided under load.** `_create_new_entity` generated IDs via
`f"entity_{int(time.time_ns())}"`. Confirmed empirically on this machine:
1000 rapid `time.time_ns()` calls returned a single identical value — clock
resolution is far coarser than nanoseconds. Genuinely distinct new entities
created in quick succession silently overwrote each other's dict entries.
This bug was masked before fix #1 (fewer `new_entity` calls happened in
quick succession when everything was over-merging via fuzzy match) and
became the dominant limiter once #1 was fixed — entity count stayed at 2-3
even after the similarity fix, until this was also fixed. Now uses
`uuid.uuid4()`, which doesn't depend on clock resolution or call rate.

After both fixes: 137 distinct entities from 148 real principals (up from
2-3), and row C's ROC-AUC recovered from worse-than-random to in-line with
baseline, with PR-AUC and precision both meaningfully improved over the raw
IF baseline. **Not a complete story yet** — ROC-AUC ~0.51 means the overall
ranking across the full score range is still close to noise, and recall at
threshold (0.114) is still low, so most individual attack events aren't
individually flagged. What's real: at the calibrated high-confidence
threshold, the system now produces a small number of alerts that are correct
more than half the time (54.5% precision, up from row A's 11.3%) — a
legitimate, defensible improvement, just not a solved detector. Further
tuning (raising `tau`, reweighting, expanding `enrichment.py`'s
`UA_PATTERNS` so tooling like `curl`/`python-requests` doesn't all collapse
to the same `"Other"` bucket) is still open — see Known gaps below.

**Known feature-engineering gaps already fixed** (see `feature_extractor.py`):
dropped `user_id` (identity leakage), `geo_country`/`is_known_proxy_or_tor`

## Final full-pipeline evaluation: IF vs XGBoost, isolated vs graph (leak-free)

Everything above used `Datasets/holdout` + `Datasets/attacks_fast`. This
section is different and important: `attacks_fast` is XGBoost's **training**
data, so it can't be reused to evaluate anything involving XGBoost's signal
without leakage. `models/evaluate_full_pipeline.py` now evaluates all four
combinations on `Datasets/holdout` + `Datasets/attacks_slow` instead — the
one dataset neither model has ever seen — for a single, consistent,
leak-free comparison. `risk_engine.py` only reads `anomaly_score` (ignoring
`phase_confidence` whenever `anomaly_score` is present — checked the source,
not assumed), so XGBoost's signal replaces `anomaly_score` for its rows,
using `1 - P(Normal)` — XGBoost's own continuous "how likely is this any
kind of attack" estimate, the same role `anomaly_score` already plays.

| Row | ROC-AUC | PR-AUC | TP @ p99 threshold |
|---|---|---|---|
| B1. IF + risk, no graph | 0.5309 | 0.0949 | 0 |
| C1. IF + risk + graph | 0.3726 | 0.0707 | 0 |
| B2. XGBoost + risk, no graph | **0.6783** | **0.1439** | 0 |
| C2. XGBoost + risk + graph | 0.5964 | 0.1120 | 0 |

Two findings, both real, neither swept under the rug:

**XGBoost's per-event signal is clearly the best-performing row (B2, ROC-AUC
0.6783)** — a real, meaningful improvement over IF (0.5309), consistent with
XGBoost's much higher standalone accuracy (0.91 vs. IF's near-random
per-event ranking).

**The graph layer hurts both signals on this test set** (C1 < B1, C2 < B2) —
the opposite of what the `attacks_fast` tuning work found. Checked whether
this is the same over-merging bug recurring: it isn't — entity resolution on
`attacks_slow` looks sane (131 distinct entities from 148 real principals,
17 genuine `fuzzy_merge`s, comparable to `attacks_fast`'s 137/148/11). The
cause is smaller-scale and not yet root-caused: even a few merges between
unrelated people (benign-with-attack or attack-with-attack-from-a-different-
campaign) can measurably dilute the risk signal when the total event count
is only ~370, since the cross-cloud multiplier and decayed-sum both operate
per-entity — a handful of contaminated entities has an outsized effect at
this scale. Not fixed yet; needs the same kind of per-entity trace-through
`attacks_fast`'s over-merging bug got, not more blind parameter tuning.

> **[Phase 1] This hypothesis was tested and DISPROVEN.** The regression is not
> caused by contaminated merges. A ground-truth merge audit found 0 mixed-label
> entities (no benign ever stitched onto an attack), and an oracle-stitching
> ablation showed that *perfect* stitching makes detection **worse**, not
> better. The real root cause is in the risk **aggregation**, not the graph —
> see "## Phase 1: root-causing the graph regression" below.

**Every row shows exactly 0 TP at its calibrated threshold, despite B2's
real 0.68 ROC-AUC — this is a threshold-calibration artifact, confirmed, not
evidence the signal is useless.** Checked directly: the p99-of-benign
threshold for B2 is 1.0497, but the single highest-scoring attack event in
the entire test set only reaches 1.0316 — a handful of unusually-scored
benign outliers (out of ~2468 calibration samples) push the 99th-percentile
bar just above every attack, even though the full-distribution ranking
(ROC-AUC) shows real separation exists. This is exactly why ROC-AUC/PR-AUC
have been the metrics to trust throughout this file, not a single fixed
operating point — `recalibrate_threshold()`'s p99 default is too fragile at
this sample size. A lower percentile (e.g. p90–p95) or a validation-set-swept
threshold (the same technique `validate_and_pick_threshold.py` already uses
for the Isolation Forest) would very likely produce a usable, non-zero
operating point here; not done yet.

> **[Phase 2] FIXED.** The threshold is no longer a benign quantile: it is
> **swept against the labeled cal set and frozen** before evaluation (shared
> `scoring_utils.sweep_threshold`, also now used by
> `validate_and_pick_threshold.py`). Every row now has a non-zero TP count at
> its frozen threshold, B2 included (0 → 144 TP on `attacks_slow`). The live
> `recalibrate_threshold()` default also moved p99 → p95 (constructor-
> configurable). See "## Phase 2: label-swept, frozen threshold" below.

## Phase 1: root-causing the graph regression (aggregation, not stitching)

Tooling added: `graph_engine.py` gained an optional `merge_audit_log` (zero
behavior change when unused) + `explain_fuzzy_similarity()`; `scoring_utils.py`
gained `ground_truth_identity()` (canonical cross-cloud actor per event) and
`discover_paths()`; `evaluate_full_pipeline.py` gained **oracle rows D1/D2**
(entity resolution keyed by ground-truth actor via the Tier-1 federation join =
perfect stitching, everything else identical to C); new script
`models/audit_entity_purity.py`. All decisions made on the **cal** set
(`holdout_cal` + `attacks_cal`), then a frozen `attacks_slow` run.

**1d — window vs lifetime (ruled out):** confirmed by unit test that the
cross-cloud multiplier is driven by the entity's **lifetime** cloud footprint,
not the 60s window (a 2nd event 120s later, alone in its window, still fires the
multiplier via `lifetime_clouds`). Not the bug.

**1b — merge purity (minor issue, not the cause):** replaying the cal set,
`audit_entity_purity.py` found **201 entities from 137 actors, 0 mixed-label**,
13 fuzzy merges of which **5 wrong**. Every wrong merge scores *exactly* `tau`
(0.65) = UA 0.45 + principal-type 0.20: unrelated GCP attack campaigns sharing
generic attacker tooling (`curl`/`python-requests`) now match on UA family+
version (informative-looking, but everyone uses them) plus the human/automation
type match. Real, but it unites attack-with-attack only — it cannot explain
C < B.

**1c — the oracle experiment (decisive):** rows D1/D2 use *perfect* stitching.
On the cal set (same shape on the frozen `attacks_slow`):

| Row (cal) | ROC-AUC | PR-AUC |
|---|---|---|
| B1. IF, no graph | 0.5298 | 0.0963 |
| C1. IF + graph | 0.3923 | 0.0899 |
| D1. IF + **oracle** graph | 0.1569 | 0.0571 |
| B2. XGB, no graph | **0.6765** | **0.1431** |
| C2. XGB + graph | 0.6064 | 0.1379 |
| D2. XGB + **oracle** graph | 0.4418 | 0.0809 |

**D ≤ C ≤ B decisively** — perfect stitching is the *worst* row. Per the plan's
decision rule, this proves stitching errors are **not** the cause; the per-
entity **aggregation** loses signal. Mechanism isolated by re-scoring the same
oracle-stitched XGB events under different `RiskEngine` configs:

| Config (oracle, XGB) | ROC-AUC | PR-AUC |
|---|---|---|
| mult=3, no baselines (= D2) | 0.4418 | 0.0809 |
| **mult=1 (cross-cloud off)** | 0.6760 | 0.1428 |
| mult=3, window 6h (not 60s) | 0.4685 | 0.0847 |
| **mult=3 + per-identity baselines** | **0.7826** | **0.4857** |

**Root cause: the cross-cloud multiplier runs with empty baselines.**
`RiskEngine.record_baseline()` is never called, so `novel_count = len(lifetime)
− 1` and every benign identity that legitimately touches 2–3 clouds over the
21-day window is amplified ×3–×9 — as much as an attacker. Perfect stitching
(D) maximizes the damage by handing every benign user their full footprint.
Disabling the multiplier fully recovers to the no-graph baseline (0.676 ≈
B2 0.677); the 60s window is a red herring (6h barely changes it). This is why
the graph *helped* on `attacks_fast`: fast attacks accumulate within the 60s
window and dominate the inflated benign scores; slow attacks never co-occur
in-window, so benign inflation wins.

**The fix (wired into the eval): baseline-aware multiplier.** Learn each
identity's normal cloud set from a **disjoint benign warmup pass** (the
`holdout` ↔ `holdout_cal` split pair from Phase 0 is exactly a same-population/
disjoint-events source for this) and `record_baseline()` before scoring, so the
multiplier only fires for clouds an identity crosses BEYOND its normal set.
`evaluate_full_pipeline.py` grew a `--baseline-dir` flag doing exactly this
(warmup pass, then `graph.reset_windows()` so no warmup event leaks into
scoring). `MultiCloudGraphEngine` gained `entity_lifetime_footprints()` +
`reset_windows()` for it.

Run it:
```bash
# calibration set: baselines from the disjoint holdout benign
python models/evaluate_full_pipeline.py --holdout-dir Datasets/holdout_cal \
    --attack-dir Datasets/attacks_cal --baseline-dir Datasets/holdout \
    --out models/ablation_results_cal_baselined.csv
# frozen test set: baselines from the disjoint holdout_cal benign
python models/evaluate_full_pipeline.py --holdout-dir Datasets/holdout \
    --attack-dir Datasets/attacks_slow --baseline-dir Datasets/holdout_cal \
    --out models/ablation_results_baselined.csv
```
(`--baseline-dir` must be disjoint from `--holdout-dir` or the script refuses —
learning baselines from the same benign you score is leakage.)

**Result — the graph layer now HELPS (frozen `attacks_slow`), C ≥ B for both
signals, and produces real TPs where p99 gave 0 before:**

| Row (frozen, +baselines) | ROC-AUC | PR-AUC | TP @ p99 (prec) |
|---|---|---|---|
| B1. IF, no graph | 0.5309 | 0.0949 | 0 |
| C1. IF + graph | **0.6299** | 0.1983 | 8 (0.24) |
| D1. IF + oracle graph | 0.7729 | 0.4804 | 65 (0.72) |
| B2. XGB, no graph | 0.6783 | 0.1439 | 0 |
| C2. XGB + graph | **0.7086** | 0.2369 | 38 (0.60) |
| D2. XGB + oracle graph | 0.7737 | 0.4111 | 89 (0.75) |

Every graph row beats its no-graph baseline (vs. C1 0.373 / C2 0.596 *without*
baselines), and oracle stitching now helps most, as a correct design should —
this vindicates the cross-cloud amplification thesis: it was never the graph
concept, only an unbaselined multiplier. **Remaining follow-on:** wire the same
`record_baseline()` (keyed by stable `entity_id`, learned from a benign warmup)
into the LIVE path — folds naturally into Phase 4's live-signal work.

## Phase 2: label-swept, frozen threshold (no more p99 fragility)

The p99-of-benign threshold was the reason every row showed 0 TP despite real
ROC-AUC. Fix, in three parts:

1. **`scoring_utils.sweep_threshold(scores, y_true, metric=...)`** — one shared
   sweep path (max-F1 or precision-floor) for BOTH the IF `decision_function`
   threshold and the risk-score `alert_threshold`. `validate_and_pick_threshold.py`
   was refactored onto it (same numbers as before: threshold −0.0776, P 0.113,
   R 0.867, F1 0.200) so the "sweep against labels, not a benign quantile"
   lesson lives in exactly one place — same discipline as `featurize_and_align`.
2. **`evaluate_full_pipeline.py`** now sweeps each row's threshold on the labeled
   **cal** set (`holdout_cal` + `attacks_cal`, baselines from `holdout`) and
   applies it **frozen** to the eval set (`holdout` + `attacks_slow`, baselines
   from `holdout_cal`) — never calibrated on the eval set. New `--cal-*` /
   `--baseline-dir` / `--cal-baseline-dir` flags; guards refuse `attacks_fast`
   and non-disjoint baseline dirs.
3. **`RiskEngine.recalibrate_threshold()`** (the live path, no labels) default
   percentile **p99 → p95**, now a constructor parameter
   (`recalibration_percentile`) with a docstring pointing at this failure mode.

**Frozen `attacks_slow`, threshold swept on cal + baselines on** — non-zero TP
everywhere, and the baseline-aware graph rows give high-precision operating
points where the no-graph rows just flood FPs:

## Risk formula note

`risk_engine.py` is a deliberately simple decayed cross-cloud risk score
(baseline + time-decayed sum of anomaly scores, multiplied by a per-new-cloud
factor), **not** a Hawkes point process. An earlier version was a Hawkes
process with MLE fitting and a goodness-of-fit test; that was intentionally
removed to keep the engine simple and easy to defend, with the point-process
version deferred until closer to the end of the project (it's still in
project history if/when it's wanted back — see the module docstring in
`risk_engine.py` for the full reasoning). The class is still importable as
`HawkesRiskEngine` for backward compatibility, but it is not currently a
Hawkes process.

## `ml_inference.py` fixed — now loads and runs both real models

`ParallelMLEngine` previously assumed a single shared 9-field raw feature
list (`_ORDERED_FEATURES`) and one generic `feature_encoder.transform(df)`
serving both models — neither existed; `app/main.py`'s loader expected files
(`isolation_forest.pkl`, `xgboost_model.json`, `feature_encoder.pkl`,
`class_names.json`) that nothing ever produced, so the app always started in
bypass mode. Rewritten to build two SEPARATELY-encoded feature vectors per
event (one per model, via `MLFeatureExtractor` reusing each model's own
fitted `label_encoders`/`feature_columns` — the same correctness reason as
`include_labeled_only_features`: the two models were trained on different
data with different appropriate vocabularies, so one shared encoder can't
serve both correctly). `app/main.py` now loads `models/iso_forest.pkl` /
`models/xgboost_classifier.pkl` as full bundles into
`state_matrix["iso_forest_bundle"]` / `state_matrix["xgboost_bundle"]`.
Verified end-to-end against a real event, not just compiled: correctly
predicted `BruteForce` at 98.9% confidence for a `ConsoleLogin` failure, with
SHAP correctly attributing it to `login_result_success` (matches that
feature's #1 overall importance rank from XGBoost training).

**Found and fixed a real crash along the way**: `models/train_xgboost.py`
trains on GPU (`device="cuda"`) by default. A GPU-resident booster combined
with `shap.TreeExplainer` crashes the process at a **native level — no
Python traceback, not a catchable exception** — reproduced directly and
isolated to exactly that combination (confirmed clean on CPU). Fixed in
`ParallelMLEngine.__init__` by switching the loaded model to
`device="cpu"` before use — free, since a single-event `predict_proba` call
has no meaningful GPU benefit anyway (GPU only helps the batch training
step, which is unaffected).

**[Phase 5] Rolling-feature train/serve skew — FIXED.** Several engineered
features (`api_call_count_1m`, `unique_ips_last_24h`, `error_rate_5m`, etc.) are
rolling/velocity features over a user's recent history; on a lone live event
they used to collapse to first-event defaults. `app/services/event_history.py`'s
`EventHistoryBuffer` now keeps a bounded trailing per-user history and
`ParallelMLEngine` featurizes each event with it, recovering the current row by
`__eval_row_id__`. See Known gaps #3 / "## Phase 2/5" and the parity test.

**[Phase 6] `xai_engine.py`'s SHAP faithfulness deletion test — FIXED.** It now
runs in engineered feature space on the `xgb_feature_row` ml_inference attaches,
deletes by replacing top-SHAP features with background reference values (not
zero), and uses a paired random-feature control. Still fails closed on error.
See Known gaps #2.
